# HomeMaster V1.2 Stage01-04 Review 报告

日期：2026-04-25  
范围：Stage01 到 Stage04 已完成链路、debug 资产、补充测试、真实 API 重跑结果。  
报告目标：说明现在做到了什么、哪里合理、哪里不合理，并用具体例子解释链路。

## 当前结论

Stage01-04 的主方向是合理的：新 `homemaster` 主链没有依赖旧 `task_brain`，任务先由 Mimo 理解成 `TaskCard`，再用 object memory 做 RAG（检索增强生成，也就是先搜记忆再给程序/模型使用），最后 Stage04 不直接相信检索 top1，而是判断有没有可靠执行记忆。

最重要的问题不是额度，也不是 BGE-M3 embedding（向量化模型，把文字转成数字向量）不可用，而是 Mimo 的结构化输出不够稳定：真实 API 重跑时，Stage01 和 Stage03 都出现过 `response_not_json_object`，也就是返回内容没有被当前 JSON 解析器稳定解析。受控探针显示，Mimo 有时会把内容放在 `thinking`（思考字段）里，或者返回 JSON 之外的额外内容。

Stage04 当前表现不错：水杯和药盒都有可靠执行记忆时能选出目标；厨房水杯已找过后，不会再选 `mem-cup-1`，而是给 Stage05 一个“未落地，需要探索寻找”的 `PlanningContext`。

## 四个阶段链路细节

### Stage01：包、CLI、契约和最小 smoke

Stage01 的作用是证明新包能跑起来，并证明 Mimo 能输出最小 `TaskCard`。例子输入是：

```text
去桌子那边看看药盒是不是还在。
```

期望输出大致是：

```json
{
  "task_type": "check_presence",
  "target": "药盒",
  "location_hint": "桌子那边",
  "needs_clarification": false
}
```

这条链路之前能通过；本次真实 API 重跑时连续两次失败，第三次通过。失败不是 HTTP 鉴权或额度问题，而是 Mimo 返回内容没有被解析成 JSON object。当前 Stage01 没有像 Stage02 那样做“非法输出后重试一次”，所以它对 Mimo 的偶发格式漂移比较敏感。

### Stage02：自然语言变成 TaskCard

Stage02 是正式任务理解入口，比 Stage01 smoke 更接近产品入口。本次真实 Mimo 四个 case 全部通过。

具体例子：

```text
去厨房找水杯，然后拿给我
```

实际 `TaskCard` 是：

```json
{
  "task_type": "fetch_object",
  "target": "水杯",
  "delivery_target": "user",
  "location_hint": "厨房",
  "needs_clarification": false,
  "confidence": 0.9
}
```

模糊指令也能正确要求澄清：

```text
帮我看看那个还在不在
```

实际输出：

```json
{
  "task_type": "unknown",
  "target": "unknown_object",
  "needs_clarification": true,
  "clarification_question": "请问您想确认哪个物品？"
}
```

这个行为是合理的：模型没有猜“那个”是什么，而是把问题交回用户澄清。

### Stage03：记忆检索

Stage03 做的是 object memory 的 RAG：Mimo 先把 `TaskCard` 改写成检索 query，然后程序用两路方法查记忆：

- BM25：词项检索，可以理解为“关键词和分词匹配”。
- BGE-M3：语义向量检索，可以理解为“意思相近也能搜到”。

水杯例子里，Mimo 生成的检索 query 是：

```json
{
  "query_text": "水杯，杯子，厨房，water cup",
  "target_aliases": ["水杯", "杯子"],
  "location_terms": ["厨房"],
  "source_filter": ["object_memory"]
}
```

检索结果排前的是：

```json
{
  "memory_id": "mem-cup-1",
  "object_category": "cup",
  "room_id": "kitchen",
  "display_text": "厨房餐桌",
  "confidence_level": "high",
  "belief_state": "confirmed",
  "ranking_reasons": [
    "bm25_rank=1",
    "dense_rank=1",
    "metadata_target_alias_match",
    "metadata_location_match",
    "metadata_high_confidence"
  ]
}
```

这说明水杯 case 是比较健康的：关键词、语义、位置、置信度都支持 `mem-cup-1`。

负证据例子里，任务仍然是找厨房水杯，但 runtime negative evidence（运行期负证据，也就是“这个地方已经找过且没找到”）包含：

```json
{
  "excluded_memory_ids": ["mem-cup-1"]
}
```

Stage03 正确把 `mem-cup-1` 放进 excluded，不让它进入可执行 hits。剩下的 `mem-cup-2` 在客厅，且是 `low` + `stale`，只能作为弱线索。

### Stage04：可靠执行记忆判定

Stage04 的关键变化是：不直接去检索 top1，而是先判断“这条记忆能不能放心作为执行目标”。

水杯 case：

```json
{
  "grounding": "grounded",
  "selected": "mem-cup-1",
  "reasons": ["reliable_execution_memory"]
}
```

含义是：`mem-cup-1` 有 memory id、房间、锚点、观察点，目标也匹配水杯，世界文件里能找到对应 viewpoint，所以可以交给 Stage05 做导航/观察计划。

负证据 case：

```json
{
  "grounding": "ungrounded",
  "selected": null,
  "assessments": [
    {
      "memory_id": "mem-cup-2",
      "status": "weak_lead",
      "reasons": ["location_conflict", "low_confidence", "stale_belief"]
    }
  ]
}
```

含义是：厨房的 `mem-cup-1` 已经被排除；`mem-cup-2` 虽然也是水杯，但在客厅、置信度低、记忆陈旧，所以不能直接导航过去当作可靠目标。Stage04 会输出：

```text
没有可靠执行记忆；Stage 05 should plan exploratory search/observe steps.
weak memory leads for exploration: 水杯 厨房
```

这正好符合我们现在的产品口径：没有找到可靠记忆不是失败，而是让 planner 先规划探索/寻找。

## Review findings

### 高优先级：真实 Mimo 输出不稳定，Stage01/Stage03 缺少同阶段重试

位置：

- `/Users/wylam/Documents/workspace/HomeMaster/src/homemaster/llm_client.py:79`
- `/Users/wylam/Documents/workspace/HomeMaster/src/homemaster/memory_rag.py:109`

现象：

- Stage01 live 第一次、第二次都失败：`response_not_json_object`。
- Stage01 第三次通过。
- Stage03 live 第一次全量是 `4 passed, 1 failed`，失败 case 是 `cup_object_memory_rag`。
- 单独重跑 cup case 通过。
- Stage03 第二次全量仍是 `4 passed, 1 failed`，失败 case 变成 `negative_evidence_excludes_location`。
- 负证据 case 单独重跑仍失败。
- 受控探针看到 Mimo 返回了 `thinking` 字段，且有时没有稳定返回可解析 JSON text。

影响：

严格验收口径下，Stage01/Stage03 真实 API 不能稳定一遍全绿。现在的工程能力可以跑通，但还不能说 live 验收完全稳定。

建议：

- 给 Stage01 和 Stage03 query 生成加一次同阶段重试，类似 Stage02 的“上一轮输出不合格，请只输出 JSON object”。
- 失败资产里保存裁剪后的 raw response，不能只写 `(no raw response captured)`。
- 如果 provider 支持，关闭 thinking 输出或强制只返回 text。
- `RawJsonLLMClient` 可以记录每次 attempt 的 raw content 摘要，但必须继续过滤密钥。

### 中优先级：Stage03 query 生成失败时不会写 FAIL debug 资产

位置：

- `/Users/wylam/Documents/workspace/HomeMaster/src/homemaster/memory_rag.py:216`
- `/Users/wylam/Documents/workspace/HomeMaster/src/homemaster/memory_rag.py:358`

现象：

Stage03 在 `query_provider.generate_query(prompt)` 这里失败时，函数直接抛出 `MemoryRagError`。这时 `actual.json` 和 `result.md` 可能仍然保留上一轮 PASS 结果。比如本次负证据 case live 失败后，磁盘上的 `negative_evidence_excludes_location/actual.json` 仍然是上一轮通过结果。

影响：

人工看 debug case 时，可能以为最后一次 live 也是 PASS；真实失败只能从 pytest 输出或 trace 里找，不够直观。

建议：

- 在 query 生成失败时也写 `actual.json` 和 `result.md`，状态标记为 FAIL。
- FAIL 报告至少包含 prompt、provider/model、错误类型、错误信息、裁剪后的 raw response。

### 中优先级：Stage04 只做房间级位置冲突，暂时不理解“桌子那边”这种锚点级提示

位置：

- `/Users/wylam/Documents/workspace/HomeMaster/src/homemaster/grounding.py:227`

现象：

药盒任务的 `location_hint` 是“桌子那边”。Stage03 检索结果里：

- `mem-medicine-1`：厨房药柜，high + confirmed，最终被 Stage04 选中。
- `mem-medicine-2`：客厅边桌，medium + confirmed，BM25/dense 原始排名更靠前，但最终分低一些。

当前 Stage04 的 location conflict 只识别厨房、客厅、储物间这类房间词。“桌子那边”不是房间，所以不会把“厨房药柜”判成冲突。

影响：

这不一定是 bug，因为“桌子那边”可能本来就模糊；但对真实家庭任务来说，用户说桌子时，系统最好至少意识到“药柜”和“桌子”不是同一种锚点。

建议：

- Stage04 可增加锚点级弱判定：如果用户明确说桌子、柜子、沙发旁，而 hit 的 `display_text/anchor_type` 明显不匹配，则降为 `weak_lead` 或在 `planning_notes` 标注不确定。
- 不建议马上写成硬拒绝，因为“桌子那边”有时只是口语范围提示。

修复后状态：

- 本轮已加入锚点级弱判定。
- “桌子那边”不再直接选“厨房药柜”；厨房药柜会因 `anchor_hint_conflict` 降为 `weak_lead`。
- 当前 Stage04 重新生成结果为：`ground_medicine_target: grounded, selected mem-medicine-2`。

## 下一步建议

Stage05 应该把 `PlanningContext` 分成两条路处理：

- `grounded`：有可靠执行记忆，可以生成导航到 `selected_target.viewpoint_id`、观察验证、后续操作的计划。
- `ungrounded`：没有可靠执行记忆，不能编造 memory id，也不能假装知道目标位置；应该先规划探索、观察或向用户追问。

建议 Stage05 的第一个测试就是：

```text
输入 PlanningContext(selected_target=None)
预期 Mimo 生成探索/观察计划，不引用不存在的 memory target。
```
